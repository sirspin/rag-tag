# Commonplace — Master Build Prompt for Orchids

## What We're Building

**Commonplace** is a collaborative newspaper platform. A small group of friends each submit links throughout the week. On a scheduled day, the EIC (Editor-in-Chief) publishes the edition — a beautifully formatted webpage that looks and feels like a real newspaper front page, with full article text where available. Contributors receive the link to read. The EIC creates the paper, sets the cadence, and invites contributors. AI organizes submissions into variably-sized sections, writes editorial ledes, extracts article text, and surfaces 2-4 recommendations based on content volume.

**V2 additions (not in scope now, but schema must support them)**: SMS delivery via Twilio, Stripe subscription gating, scheduled auto-compilation, browser extension, print tier, custom domains.

---

## Tech Stack

- **Framework**: Next.js 14 (App Router)
- **Database + Auth**: Supabase (magic link auth, no passwords anywhere)
- **Article extraction**: Mozilla Readability (via @mozilla/readability + jsdom) — server-side, extract at compile time
- **AI**: Claude API (claude-sonnet-4-20250514) for section organization, editorial ledes, and recommendations
- **OG Metadata**: Auto-fetch title, description, image from submitted URLs
- **Hosting**: Vercel

- `POST /api/editions/publish` — Set edition to published
- `POST /api/invites/send` — Send invite email

## Article Extraction Logic

At compile time (when EIC hits "Compile Edition"), for each submission:

1. Fetch the URL server-side
2. Run Mozilla Readability on the HTML via jsdom
3. **If extraction succeeds** (returns >200 words of clean text):
   - Store extracted text in `submissions.extracted_text`
   - Set `extraction_status = 'success'`
   - Truncate display to a natural paragraph break around 400-600 words
   - Show "Continue reading at [og_site_name] →" at truncation point
4. **If extraction returns thin content or fails**:
   - Attempt to detect paywall indicators in the HTML
   - If paywalled: set `extraction_status = 'paywalled'`, display OG card + "Read the full story at [Source] →"
   - If just failed: set `extraction_status = 'failed'`, same fallback treatment
5. Pass extracted text excerpts to Claude for richer AI organization

**Important**: Do not cache extracted_text permanently beyond the edition lifecycle. Extract at compile time, store through the edition, but this is not an article archive. The edition is the artifact.

**Fallback display treatments:**
- `success`: Article text rendered in edition, truncated at natural break, with italic "Continue reading at [Source Name] →" — styled like a newspaper "Continued on B2"
- `paywalled`: OG image + headline + OG description + "Read the full story at [Source Name] →"
- `failed`: Same as paywalled

## AI Compilation (Claude API)

Called after article extraction completes. Send to Claude:

**System prompt:**
```
You are the editorial AI for Commonplace, a collaborative newspaper platform.
Your job is to turn a group of friends' submitted links into a beautifully organized newspaper edition.

Rules:
- Create 2-5 sections with evocative, single-word or short titles (e.g. "Culture", "On Our Minds", "The Long Read", "Noise", "Signal")
- Write a 1-2 sentence editorial lede for each section that feels authored — observe what the section reveals about the group's collective reading that week. Be dry, warm, occasionally wry. Never generic.
- Assign each section a weight: "lead" (1-2 strong submissions, gets full-width treatment), "standard" (2-3 submissions, column grid), or "brief" (1 short item, runs narrow). Every edition should have a different combination of weights so no two editions look identical. Maximum one "lead" section per edition.
- Supply 2-4 article recommendations based on themes present. More recommendations if the edition is thin on submissions, fewer if rich.
- Order submissions within sections by editorial flow, not submission order.

Return ONLY valid JSON. No preamble, no markdown fencing.
```

**User message:**
```json
{
  "paper_name": "string",
  "edition_number": "int",
  "submissions": [
    {
      "id": "uuid",
      "url": "string",
      "og_title": "string",
      "og_description": "string",
      "og_site_name": "string",
      "note": "string or null",
      "contributor_display_name": "string",
      "extracted_text_excerpt": "first 300 words of extracted text, or null"
    }
  ]
}
```

## The Edition Page — Most Important Screen

This is the product. Every design decision serves one goal: it should feel like holding a real newspaper.

### Visual Design

**Aesthetic**: Heritage editorial print. Letterpress broadsheet — cream paper, heavy black ink, column grids, thin rules. References: the newspaper "Origen" (cream background, bold serif masthead, #0001 edition badge, column grid layout), classic broadsheet structure, Semafor's editorial confidence.

**Color palette:**
```css
--background: #F5F0E8;       /* warm cream, newsprint */
--text-primary: #1A1209;     /* near-black, warm undertone */
--text-secondary: #5C4A2A;   /* warm mid-brown, bylines and metadata */
--rules: #2A1F0E;            /* dark ink brown, column dividers */
--accent: #8B0000;           /* deep red, edition badge and section accents */
--link-color: #1A1209;       /* same as text — underline only on hover */
```

**Typography (Google Fonts):**
- Masthead (paper name): Playfair Display Black — very large, slightly tracked
- Section headlines / article headlines: Playfair Display or Lora
- Body text / extracted article text: Lora — readable at small sizes
- Bylines: EB Garamond Italic
- Section headers: EB Garamond small-caps
- Edition metadata / badge: Courier Prime monospace

**Texture**: Subtle CSS noise grain overlay on background via SVG filter pseudo-element. Should feel like paper, not a screen.

### Layout Weight Rules

- `lead`: Full-width section. Large headline, full OG image, article text if available. Maximum one per edition.
- `standard`: Two-column grid. OG image, headline, byline, note, text excerpt or OG description.
- `brief`: Single narrow column. Headline + byline + source link only. No image.

AI assigns weights — layout grid responds to them. Every edition looks structurally different.

**Mobile**: Single column throughout. Masthead stays large and centered. Section headers and rules intact. OG images full-width above headline. Article text fully readable.

### Article Text Rendering

- Extracted text rendered in Lora, body size, proper paragraph spacing
- Truncated at natural paragraph break, 400-600 words
- Truncation line: `*Continue reading at [Source Name] →*` — italic, warm brown, right-aligned at column bottom
- Paywalled/failed: OG description rendered instead, followed by `*Read the full story at [Source Name] →*`

## Invite Flow

1. EIC enters contributor email
2. System creates invite record + sends email
3. Email subject: "[EIC display name] wants you to write for [Paper Name]"
4. Email body: brief explanation + "Claim your byline" CTA → `/invite/[token]`
5. Invite page: shows paper name, asks for display name (byline), confirms email
6. Magic link sent → on sign-in, membership activated, redirect to submission page

**Critical**: No "create an account" screen. The invite page IS onboarding.

- Write like a publication, not a SaaS app
- No "workflow", "leverage", "streamline", "platform"
- Edition statuses: "In progress" / "Ready to compile" / "Published"
